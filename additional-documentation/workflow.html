Deploying to Firebase Hosting

The Calvin EMR web application is hosted live on Firebase Hosting at https://calvin-emr.web.app. The app is configured with Firebase's GitHub Action feature for continuous deployment. The live site is built and deployed off of the main branch in our GitHub repository. Firebase also generates a live preview site for each pull request that is made. It is recommended to look at the pull request's preview before merging into the main branch, since the live preview shows what the actual site will look like after merging.

After a branch is merged into main, Firebase automatically builds and deploys the updated application.

You can read more about Firebase Hosting here.

Deploying Cloud Functions

The application uses Firebase Cloud Functions to handle user account creation and deletion. You can read more about Firebase Cloud Functions here.

To deploy the Cloud Functions, run the following command:

firebase deploy --only functions

This will run a linter on the files containing the function code to ensure the code is formatted correctly. This check may fail, so be sure to fix all errors that are displayed in the console before running the command again (luckily, the errors are pretty informational).

After the code passes the formatting test, it will be packaged and deployed to Firebase. You can view the functions and the function logs under the Functions tab in the Firebase project.

Generating Documentation

The documentation that you are currently looking at was generated using Compodoc. All files are located in the documentation and markdown directories.

Building the Documentation Site

The documentation is compiled manually via npm scripts. The scripts are described below:

  • npm run compodoc:build: builds (or rebuilds) the static site that contains all documentation. Stores the site's files into a folder named documentation. Documentation is generated based on (1) JSDoc comments in the source code and (2) Markdown files in the markdown directory. See below for more information on both directories.

  • npm run compodoc:serve: serves the generated documentation site on the local machine.

  • npm run compodoc:build-and-serve: builds and serves the documentation site in sequence.

Important: If you have changed and built the documentation in a separate branch off main and are looking to make a pull request, make sure to delete the documentation folder from your branch before creating a pull request. Otherwise, the pull request will generate several merge conflicts that take significant time to resolve. To prevent this from happening:

  1. Delete the documentation directory from your branch.
  2. Create and merge the pull request.
  3. Run npm compodoc:build to build your new documentation in the main branch.

documentation Directory

The static site is served from this directory. This directory and its files are generated each time npm run compodoc:build and npm run compodoc:build-and-serve are run. The two commands rebuild the documentation site based on the JSDoc comments and tags in the src folder, as well as Markdown files specified in the markdown directory. As a result, do not place any additional files in this directory. They will be overwritten the next time the documentation is generated.

If you need to extend the documentation, reference the markdown directory section below.

markdown Directory

This is the directory to put all .md files that you wish to include under the Additional Documentation tab in the final static site. This directory contains the summary.json file, which specifies which Markdown files to compile and display in the final documentation. All of the .md files listed in summary.json should be placed in the markdown directory.

You can read more about using the Additional Documentation tab here.
